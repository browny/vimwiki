Diagnose: TRUE

Transparent: The consequence of change are visible and predictable

Reasonable: The cost of adding a new feature is proportional to its value

Usable: If you already wrote the code, you can reuse it

Exemplary(模範): More code like this would be good for your app

Dependencies: It exists -> What does it do -> How does it do that

Dependencies 是不可避免的 -> Rule: Depend on things more stable than you are.

如何判斷什麼是比較 stable 的 -> you don't know!! 在設計的初期,並沒辦法輕易的看出,而且會隨著 buiness model change 而改變

Dependencies: It exists (代表你預期他會是什麼樣的物件, 根據接收的物件來判斷要呼叫的方法是危險的, depend on abstraction, not implementation) -> What does it do (Just tell what it should do, not to call its method) -> How does it do that (封裝, 僅開放外部需要的 method, 內部實際的 implementation 不對外透漏)

Abstractions are more stable than concretions!

Follow the message, Don't be stuck on the Class. Create the Abstraction. Embrace the Abstraction.
